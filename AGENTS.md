# 프로젝트 에이전트 가이드 (AGENTS.md)

이 문서는 AI가 이 저장소에서 코드를 작성·수정할 때 반드시 따를 규칙입니다. 매 세션마다 같은 설명을 반복하지 않도록 여기에 스택과 원칙을 고정합니다.

## 기술 스택 (고정)

| 영역 | 선택 |
|------|------|
| 프레임워크 | **Next.js**, **App Router** (`app/` 디렉터리) |
| 언어 | **TypeScript** (`.ts` / `.tsx`만 사용, 새 파일에 순수 JS 금지) |
| 스타일 | **Tailwind CSS** (유틸리티 클래스 우선) |
| 데이터베이스·백엔드 | **Supabase** (클라이언트·서버 클라이언트·RLS 정책 준수) |

---

## 해야 할 것 (DO)

- **App Router 규칙**: 라우트는 `app/` 아래에 두고, `page.tsx`, `layout.tsx`, `loading.tsx`, `error.tsx`, `route.ts` 등 App Router 관례를 따른다.
- **서버 vs 클라이언트**: 브라우저 API·`useState` 등이 필요하면 파일 상단에 `'use client'`를 명시하고, 기본은 서버 컴포넌트를 우선한다.
- **TypeScript**: 공개 API·props·DB 행 타입 등에 명시적 타입을 둔다. `any`는 정말 불가피할 때만 최소 범위로 사용한다.
- **Tailwind**: 스타일은 Tailwind 유틸리티로 맞추고, 재사용이 많은 패턴은 `@apply`보다 **컴포넌트 추출** 또는 **공통 className 상수**를 우선한다.
- **Supabase**: 환경 변수(`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` 등)로만 키를 주입하고, **서비스 롤 키는 클라이언트 번들에 넣지 않는다.** RLS를 전제로 하고, 서버 전용 작업은 Route Handler나 Server Action 등 **서버 측**에서 서비스 롤을 사용한다.
- **데이터 접근**: 사용자별 데이터는 RLS와 인증 상태를 고려해 설계하고, 민감한 쿼리는 서버에서만 수행한다.
- **일관성**: 기존 파일의 import 순서, 네이밍, 포매터(프로젝트에 설정된 경우)에 맞춘다.

---

## 하면 안 되는 것 (DON'T)

- **Pages Router를 새로 도입하지 않는다** (`pages/`에 새 라우트 추가 금지). 기존 레거시가 없다면 `pages/`를 만들지 않는다.
- **새 파일을 `.js` / `.jsx`로 추가하지 않는다.**
- **CSS-in-JS로 전체 스타일을 갈아엎거나**, Tailwind 없이 인라인 스타일만으로 화면을 구성하지 않는다. (필요한 소량 인라인은 예외 가능)
- **Supabase 서비스 롤 키를** 클라이언트 컴포넌트, `NEXT_PUBLIC_*` 변수, 또는 브라우저로 노출되는 코드에 넣지 않는다.
- **RLS를 끄거나 우회하는 방식**을 기본 해결책으로 제안하지 않는다. (디버그 한정이라도 문서화·최소 권한을 명시한다.)
- **시크릿을 코드에 하드코딩하지 않는다.**
- 불필요한 **대규모 리팩터**나 요청 범위 밖 파일 변경을 하지 않는다.

---

## 참고

- 의존성 추가가 필요하면 `package.json`과 프로젝트에 이미 쓰는 패턴을 확인한 뒤, 한 가지 방식으로 통일한다.
- 이 문서와 충돌하는 지시가 있으면 **이 문서를 우선**하고, 예외가 필요하면 사용자에게 확인한다.
